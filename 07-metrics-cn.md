# 维度七：智能体性能指标（Metrics & Thresholds）

> 回答：怎样衡量这个 Skill 做得好不好？什么时候需要改进或重构？

---

## 核心指标定义

| 指标 | 定义 | 收集方式 |
|------|------|----------|
| **模型调用量** | 单次任务消耗的 token 总数 | API 日志 |
| **回答采纳率** | 用户采纳/使用 Skill 输出结果的比例 | 人工记录或反馈 |
| **运行时错误率** | L3 致命错误出现频率 | Error.md 统计 |
| **知识库命中率** | 知识库条目被成功引用的次数 | 知识库元数据 |
| **执行时长** | 单次任务从启动到交付的总耗时 | log.md 时间戳 |

---

## 量化基准与触发阈值

| 指标 | 绿区（正常） | 黄区（预警） | 红区（需干预） | 触发动作 |
|------|------------|------------|------------|---------|
| **采纳率** | ≥ 85% | 60% ~ 84% | < 60% | 黄：优化 Skill.md → 红：评审 Skill 设计 |
| **L3 错误率** | < 2% | 2% ~ 5% | > 5% | 黄：更新 Error.md 预防建议 → 红：触发 Skill 重构评审 |
| **执行时长** | < 5 分钟 | 5 ~ 15 分钟 | > 15 分钟 | 黄：分析瓶颈 → 红：考虑拆分为子 Skill |
| **知识库命中率** | > 70% 查询有结果 | 40% ~ 70% | < 40% | 黄：补充知识库 → 红：重构知识库结构 |
| **Token 消耗** | 符合预期 | 预期 × 1.5 | 预期 × 3 | 黄：优化 Prompt → 红：评估模型选型 |

---

## 知识库升级与降级规则

> 以下是**后台机制**，不需要显式写入操作步骤，但在 Skill.md 中声明：

```yaml
knowledge_base:
  hit_rate_threshold: 0.7    # 命中率低于此值触发知识库审查
  upgrade_rule:
    # 外部获取的新知识 → 自动建档，标注来源和日期
  downgrade_rule:
    # 连续 30 天未被引用的条目 → 移入「归档知识库」
    # 归档知识库 90 天后可删除
```

## 性能数据记录

| 记录项 | 存储位置 | 更新频率 |
|--------|----------|----------|
| 采纳率 | `~/.workbuddy/skills/<skill>/metrics.csv` | 每次用户反馈时 |
| 错误统计 | `~/.workbuddy/skills/<skill>/errors.csv` | 每次 Error.md 生成时 |
| 执行时长 | 同上 metrics.csv | 每次 log.md 生成时 |
| 知识库引用 | `~/.workbuddy/skills/<skill>/wiki/index.json` | 每次知识查询后 |

---

## 改进触发流程

```
指标进入黄区
  │
  ▼
分析根因（查看 metrics.csv + Error.md）
  │
  ▼
制定改进计划（更新 Skill.md / 知识库 / 基础设施）
  │
  ▼
在 Skill.md changelog 中记录本次改进
  │
  ▼
小范围测试 → 采纳率恢复至绿区？→ 正式发布新版本
```
